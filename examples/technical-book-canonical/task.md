# Task

## Incoming request

> Use the task graph framework to help an AI agent write a 20-chapter technical book. The project already has governance, standards, requirements, and chapter specifications. The goal is high quality, not bulk output for its own sake.

## What already exists

- a governance packet
- chapter specifications
- style and pedagogy requirements
- diagram expectations
- quality-sensitive reader expectations

## What tends to go wrong without a graph

In this class of task, an unconstrained AI often fails in predictable ways:
- chapters start strong and then become templated
- later chapters shrink in depth or ambition
- standards adherence becomes uneven
- diagrams degrade or become generic
- revisions are attached as addenda instead of being integrated
- local chapter polish hides whole-book weakness

## What the graph must solve

The graph must not only schedule work. It must decide:
- what the real execution unit is
- whether bulk generation is currently safe
- whether a pilot sample must be accepted first
- which quality-review nodes are mandatory
- when to fan out into parallel chapter work
- when to shrink back if quality degrades

## Resolved task classification

- **Primary type:** writing / content-heavy
- **Secondary type:** diagram-rich educational work
- **Subjectivity:** high
- **External-fact dependence:** low
- **Scale risk:** high
- **Merge sensitivity:** high

## Resolved fundamental unit

The framework selects **chapter** as the fundamental unit.

This is because a chapter is:
- large enough to reveal the real quality bar
- meaningful enough to judge pedagogy and tone
- small enough to revise before unsafe scale-out
- the right unit for deciding whether parallel creation is safe

## Immediate implication

Bulk multi-chapter generation is blocked at the start.

The graph must first create and review a representative **pilot sample chapter**.
