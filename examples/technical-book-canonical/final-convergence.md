# Final Convergence

This page explains what happens after chapter workers have produced multiple chapters.

At this stage, the framework shifts from **chapter quality** to **book quality**.

That is a critical transition.

## Why final convergence matters

A system can produce several individually strong chapters and still fail as a book.

Typical whole-book failure modes:
- repeated rhythms across chapters
- uneven chapter depth
- diagram system inconsistency
- inconsistent adherence to chapter contracts
- drift away from the approved pilot shape
- local fixes that damage global coherence

That is why the graph inserts convergence logic rather than jumping from chapter creation straight to completion.

## Main nodes in convergence

### 1. MERGE_REVIEW
This checks the outputs before harmonization.

It looks for:
- conflicting assumptions between chapters
- repeated patterns that became too regular
- chapter-family divergence
- interface surfaces such as shared terminology, repeated concepts, or visual system mismatches

### 2. HARMONIZATION
This turns multiple chapter outputs into one coherent family.

It may:
- normalize conventions
- align recurring terminology
- reduce repeated explanatory scaffolds
- smooth chapter-to-chapter transitions
- standardize visual / caption behavior

### 3. WHOLE_SYSTEM_COHERENCE_QA
This is where the framework checks the book as a single artifact.

It asks:
- does the book progression work?
- are chapter depths balanced?
- did later chapters degrade?
- is the visual system coherent?
- are standards being followed consistently across the whole manuscript?

## Shrink-back behavior

One of the framework’s strongest features is that it does not assume scaling is permanent.

If whole-system review finds that quality has degraded, the graph may shrink back.

### Example shrink-back actions
- stop opening new chapter workers
- return to single-chapter calibrated execution
- regenerate later chapter prompts from the stronger pilot doctrine
- patch only after the chapter shape is stable again

This is different from naïve bulk workflows, which often keep going after visible degradation simply because momentum has already started.

## What successful convergence looks like

A successful final convergence phase should produce:
- a coherent chapter family
- controlled variation without templated sameness
- consistent standards adherence
- diagrams that support the same teaching system
- a stable book-wide quality bar

## Final result of the example

This example demonstrates that the framework does more than generate a graph.

It also governs:
- when scaling begins
- how workers are constrained
- how outputs are merged
- when the system should retreat to a smaller execution unit again

That is what makes the framework useful for quality-sensitive long-form projects.
